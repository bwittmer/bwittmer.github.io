Capstone Project

For my senior year Capstone project, my group was tasked with making a web-based application for the faculty of Ball State University. The purpose of this app is, to make it easier for members to loan and borrow Realia items.

Tech-Stack

  • PHP - The language the whole app is programed in
  • CakePHP - A PHP framework that allows for easier back- and front-end PHP development
  • Amazon Web Services ( AWS ) - The host for the app
  • MySQL - The database technology
  • PHPStorm - The IDE that the app was programed in
  • GitHub - The host of the code repositories

My Contributions

I was the primary back-end developer of this application. A brief overview: I refactored much of the existing code, I implemented tests for the software, I restructured the SQL-Database, and I was the developer who conceptualized and implemented the messaging system that Regalia Share employs.

Refacotred Code

This project was not originally ours, so when we came in to start working on it, we were all surprised at the complexity of this project, however, it was only conceived complexity. I am a very much hands-on learner, so the way I that I became familiar with this software was by looking at the code and refactoring it.

This page's backend was heavily refactored, to introduce more OOP principles, one of which is the principle of cohesion, where each function does one thing, and one thing only.

This page's backend was completely reworked. Originally, the 'Cancelled' field was in another field within the Reservation entity, and I merged the tow fields, which were `Reservation_approval` and `Reservation_Status`. This allows for cleaner query requests and allows for a streamlined experience without the extra clutter of more fields.

Newly Implemented Code

After I became familiar with the code, I was the main force in conceptualizing and implementing a messaging system for the end-users.

The development of this functionality went very smoothly as I was able to draw on my past experiences with the refactoring that I did. Messages is tied to each reservation, a new message is created each time that a reservation is approved, and then within the back end all the important information is stored till the new message is sent.

Updating the MySQL

I was the primary SQL developer, and I was tasked with updating the SQL to fit our needs within the scope of the project. There was a lot of deprecated features that had to be removed in entirety, and the SQL had to reflect that. As well as adding a new table for the new messaging feature.

Testing

When we were originally joined this project, there were only about 50 tests, and not everything was tested. Staring in our second iteration, I began to refactor the tests that were included to make all tests pass and preform the newer functionality. During the 3rd iteration I went through and added new tests to the existing code base, and we ended up with 160 tests. The 4th iteration was when I went through and added the remaining tests, tests for the Messaging feature, and removed all tests that were no longer needed.
